# Phase 7. 대용량 업로드 전략

## 목적

이 문서는 Cathy Garden의 업로드 전략을 정리한다.

현재 목표는 다음 두 가지다.

- 여러 개의 사진/영상을 한 번에 업로드할 수 있어야 한다.
- 큰 파일, 특히 긴 동영상 업로드에 대비한 다음 단계 전략을 미리 정해 둔다.

## 현재 구현 원칙

현재 구현은 `presigned PUT URL` 기반 직접 업로드를 사용한다.

흐름:

1. 브라우저가 업로드할 파일 정보를 서버에 보낸다.
2. 서버가 S3 presigned URL을 발급한다.
3. 브라우저가 S3로 직접 업로드한다.
4. 업로드 완료 후 메타데이터를 저장한다.

이 구조는 앱 서버를 파일 중계 경로에서 제외하므로, 개인 프로젝트 기준 비용과 단순성 면에서 적절하다.

## 여러 파일 업로드 전략

여러 파일 업로드는 우선 `동시 업로드`보다 `순차 업로드`를 기본으로 둔다.

이유:

- 모바일 네트워크에서 실패 지점이 단순하다.
- manifest 갱신 충돌을 줄이기 쉽다.
- 브라우저 메모리와 네트워크 부하를 낮게 유지할 수 있다.
- 진행 상태 문구를 단순하게 유지할 수 있다.

현재 단계에서의 기본 전략:

- 파일 여러 개 선택 허용
- 각 파일을 순서대로 presign
- 각 파일을 순서대로 S3 업로드
- 각 파일 업로드 후 즉시 메타데이터 저장
- 한 파일이 실패하면 배치 전체를 멈추고 오류를 보여준다

## 대용량 파일 전략

현재 단일 presigned PUT 업로드는 중간 크기 파일에는 충분하지만, 긴 동영상처럼 더 큰 파일에는 한계가 있다.

다음 단계 전략은 `S3 multipart upload`다.

multipart가 필요한 이유:

- 대용량 파일 업로드 안정성 향상
- 네트워크 끊김 시 전체 재업로드 대신 일부 파트 재시도 가능
- 모바일 환경에서 긴 영상 업로드에 유리

## 단계별 적용 계획

### 현재 단계

- 사진/영상 여러 개 선택 가능
- 순차 업로드
- 업로드 완료 후 보관함 반영

### 다음 단계

- 파일 크기 기준으로 multipart 대상 분기
- multipart 시작 API
- part presign API
- complete multipart API
- 실패 파트 재시도

## 권장 기준

현재 문서 기준으로는 다음처럼 운영한다.

- 일반 사진과 짧은 영상: 현재 presigned PUT 유지
- 큰 영상: 차기 작업에서 multipart로 전환

즉, “여러 개 업로드”와 “대용량 multipart”는 같은 업로드 축에 있지만, 구현 단계는 분리한다.

## 완료 기준

아래 조건을 만족하면 현재 단계 목표를 달성한 것으로 본다.

- 사용자가 여러 파일을 한 번에 선택할 수 있다.
- 선택된 파일이 순차적으로 S3에 올라간다.
- 업로드 완료 후 보관함에 반영된다.
- 모바일에서도 상태 문구와 성공 흐름이 유지된다.
- multipart 도입 대상과 다음 API 범위가 문서로 정리된다.
