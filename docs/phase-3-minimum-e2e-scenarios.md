# Phase 3. 최소 E2E 시나리오

## 목적

이 문서는 Cathy Garden에서 반드시 보장해야 하는 최소 end-to-end 시나리오를 정의한다.

여기서 말하는 E2E는 실제 사용자 입장에서 브라우저를 통해 기능을 끝까지 수행하는 흐름을 의미한다.

이 문서의 목표는 다음과 같다.

- 배포 전 반드시 확인해야 할 핵심 흐름을 고정한다.
- Playwright 같은 도구로 자동화할 때 기준 시나리오를 제공한다.
- 단위 테스트와 통합 테스트만으로 놓치기 쉬운 연결 구간을 보완한다.

## 현재 제품 범위 기준

현재 구현 기준으로 최소 E2E는 아래 다섯 축을 포함해야 한다.

1. 로그인
2. 업로드
3. 보관함 반영
4. 상세 보기
5. 삭제

이 다섯 축이 통과하면 현재 MVP의 핵심 사용자 흐름은 살아 있다고 볼 수 있다.

## 시나리오 1. 비로그인 사용자는 보호 페이지에 들어갈 수 없어야 한다

### 목적

- 보호 라우트가 실제 브라우저 흐름에서도 동작하는지 확인

### 절차

1. 비로그인 상태로 `/library` 접근
2. 로그인 페이지로 이동되는지 확인
3. `/upload` 접근
4. 다시 로그인 페이지로 이동되는지 확인

### 기대 결과

- 보호 페이지는 직접 열 수 없어야 한다.
- 로그인 화면이 표시되어야 한다.

## 시나리오 2. 올바른 비밀번호로 로그인 후 보관함에 진입할 수 있어야 한다

### 목적

- 실제 로그인 폼과 세션 쿠키 흐름 확인

### 절차

1. `/login` 접근
2. 올바른 `CATHY_GARDEN_PASSWORD` 입력
3. 로그인 제출
4. `/library` 또는 기본 보호 페이지로 이동되는지 확인

### 기대 결과

- 로그인 성공 후 보호 페이지에 진입한다.
- 헤더에서 private route 링크가 보인다.

## 시나리오 3. 잘못된 비밀번호는 거절되어야 한다

### 목적

- 인증 실패 흐름과 사용자 피드백 확인

### 절차

1. `/login` 접근
2. 잘못된 비밀번호 입력
3. 로그인 제출

### 기대 결과

- 로그인에 실패한다.
- 로그인 페이지에 머문다.
- 오류 메시지가 표시된다.

## 시나리오 4. 사진 여러 장을 업로드하고 보관함에 반영할 수 있어야 한다

### 목적

- 다중 업로드의 핵심 경로 확인
- presign -> S3 업로드 -> metadata 저장 -> library 반영까지 실제 연결 확인

### 절차

1. 로그인 상태에서 `/upload` 접근
2. 허용된 사진 파일 2개 이상 선택
3. 업로드 시작
4. 성공 메시지 확인
5. 자동 이동 또는 수동 버튼으로 `/library` 이동

### 기대 결과

- 선택한 파일 수만큼 업로드가 완료된다.
- 업로드 완료 배너가 표시된다.
- 보관함 목록에 새 항목이 보인다.

## 시나리오 5. 사진 상세 페이지에서 원본 미리보기를 볼 수 있어야 한다

### 목적

- 보관함 카드 -> 상세 페이지 -> signed preview 연결 확인

### 절차

1. 보관함에서 업로드된 사진 카드 선택
2. 상세 페이지 진입
3. 이미지 미리보기가 렌더되는지 확인

### 기대 결과

- 상세 페이지가 열려야 한다.
- signed URL 기반 이미지가 표시되어야 한다.

## 시나리오 6. 동영상 업로드 후 상세 페이지에서 재생할 수 있어야 한다

### 목적

- 영상 업로드와 재생 경로 확인

### 절차

1. 로그인 상태에서 동영상 파일 업로드
2. 보관함에서 영상 카드 선택
3. 상세 페이지 진입
4. `<video>` 플레이어 렌더 확인

### 기대 결과

- 동영상 카드가 보관함에 나타난다.
- 상세 페이지에서 플레이어가 보여야 한다.

## 시나리오 7. 보관함에서 필터와 정렬이 실제로 동작해야 한다

### 목적

- 사진/영상 필터와 최신순/오래된순 정렬 확인

### 절차

1. 사진과 동영상이 섞인 상태 준비
2. `Photos` 필터 선택
3. 사진만 보이는지 확인
4. `Videos` 필터 선택
5. 영상만 보이는지 확인
6. `Newest first`와 `Oldest first` 전환

### 기대 결과

- 필터 결과가 정확해야 한다.
- 정렬 기준이 바뀌면 순서가 달라져야 한다.

## 시나리오 8. 상세 페이지에서 삭제하면 보관함에서 사라져야 한다

### 목적

- 원본 삭제 + manifest 갱신 + 목록 반영까지 확인

### 절차

1. 상세 페이지에서 `Delete` 클릭
2. 브라우저 confirm 수락
3. 보관함으로 이동
4. 삭제한 항목이 더 이상 없는지 확인

### 기대 결과

- 삭제 요청이 성공해야 한다.
- 보관함에서 해당 항목이 사라져야 한다.

## 자동화 우선순위

Playwright 기준 자동화 우선순위는 아래 순서로 둔다.

### 1순위

- 시나리오 1
- 시나리오 2
- 시나리오 4
- 시나리오 8

이 네 개가 현재 MVP의 최소 통과선이다.

### 2순위

- 시나리오 3
- 시나리오 5
- 시나리오 7

### 3순위

- 시나리오 6

동영상은 파일 준비와 실행 시간 때문에 약간 뒤로 둔다.

## 테스트 데이터 원칙

- 사진 fixture 2장
- 동영상 fixture 1개
- 크기가 작은 샘플 파일을 사용
- E2E는 운영 버킷이 아닌 테스트용 prefix 또는 테스트 버킷 사용 고려

예시:

- `uploads/e2e/...`
- 또는 별도 테스트 버킷

## 환경 준비 조건

E2E 자동화 전에 아래가 준비되어 있어야 한다.

- `CATHY_GARDEN_PASSWORD`
- `AWS_REGION`
- `AWS_S3_BUCKET`
- `AWS_S3_MEDIA_MANIFEST_KEY`
- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- 업로드 도메인 기준 S3 CORS 설정

## 현재 단계의 완료 기준

아래 조건을 만족하면 `최소 E2E 시나리오 정의`가 완료된 것으로 본다.

- 핵심 사용자 흐름이 시나리오로 문서화되었다.
- 우선순위가 정리되었다.
- fixture 및 환경 준비 조건이 정리되었다.
- 추후 Playwright 자동화로 옮길 수 있는 수준으로 구체화되었다.
