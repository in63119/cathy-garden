# Phase 2. 기술 전환 설계

## 목적

이 문서는 현재 Cathy Garden 저장소를 기존 구조에서 개인용 사진/영상 보관함 구조로 전환하기 위한 기술 설계를 정리한다.

Phase 1에서 제품 방향은 확정되었고, Phase 2에서는 그 제품을 어떤 기술 구조로 구현할지 결정한다.

## 이번 단계에서 확정된 사항

- 배포 환경은 `Vercel`로 간다.
- 프론트와 서버는 `Next.js` 한 프로젝트로 통합한다.
- 파일 저장소는 `Amazon S3`를 사용한다.
- 현재 프로젝트의 블록체인 관련 요소와 `contract/` 워크스페이스는 제거 대상이다.

## 현재 구조 요약

현재 저장소는 아래 두 축으로 나뉘어 있다.

- 루트: `Create React App + CRACO + Recoil + MUI`
- `contract/`: `Hardhat + OpenZeppelin + Typechain`

현재 문제는 다음과 같다.

- 프론트엔드와 백엔드가 통합되어 있지 않다.
- 인증 처리가 브라우저 중심이다.
- 사진/영상 업로드를 위한 서버 구조가 없다.
- 스토리지 연동 구조가 없다.
- 현재 UI 구조가 NFT 방향의 흔적을 아직 포함하고 있다.
- `contract/`는 현재 제품 목표와 무관하다.

## 목표 아키텍처

최종적으로 지향하는 구조는 다음과 같다.

```text
Next.js App
├── App Router
├── Server Routes
├── Auth Layer
├── Upload API
├── Media Library UI
└── S3 Integration
```

구성 요소별 역할:

- `Next.js`
  - 프론트엔드 화면
  - 서버 라우트
  - 인증 처리
  - 업로드 서명 API
- `Vercel`
  - 애플리케이션 배포
  - 환경변수 관리
- `Amazon S3`
  - 실제 사진/동영상 원본 저장
- `Database`
  - 사용자 정보, 파일 메타데이터, 정렬/조회 정보 저장

## 권장 기술 스택

### 프레임워크

- `Next.js`
- `TypeScript`
- `App Router`

선정 이유:

- 프론트와 서버를 한 저장소 안에서 같이 운영할 수 있다.
- `Vercel`과 배포 궁합이 좋다.
- 인증, 업로드 서명 API, 파일 목록 API를 한 코드베이스에서 처리할 수 있다.

### UI

- 기본은 `React` 기반
- 스타일링은 전환 시점에 재선택
- 현재 `MUI`는 유지 여부를 별도로 판단

현재 기준 판단:

- 기존 MUI를 억지로 끌고 가기보다, 새 디자인 시스템에 맞게 재구성하는 편이 낫다.
- 전원주택/정원 중심 비주얼을 위해서는 범용 대시보드 스타일보다 커스텀 UI 비중이 높아질 가능성이 크다.

### 상태 관리

- 전역 상태는 최소화한다.
- 인증/서버 데이터는 가능한 한 서버 기준으로 관리한다.
- 클라이언트 전역 상태 라이브러리는 꼭 필요할 때만 도입한다.

판단:

- 현재 `Recoil`은 전환 시 제거하는 쪽이 적절하다.
- 서버 데이터는 향후 필요 시 `TanStack Query`를 검토할 수 있다.

### 스토리지

- `Amazon S3`

선정 이유:

- 표준적이고 안정적인 오브젝트 스토리지다.
- 문서와 예제가 풍부하다.
- presigned URL 기반 업로드 구조를 적용하기 쉽다.
- 사진/동영상 원본 저장 구조에 적합하다.

### 데이터베이스

후보:

- `PostgreSQL`

판단:

- 업로드된 파일 목록, 메타데이터, 접근 제어를 안정적으로 다루려면 DB 도입이 필요하다.
- 초기에는 단순한 스키마로 시작하고, 이후 확장 가능하게 유지한다.

### ORM

후보:

- `Prisma`
- `Drizzle`

현재 권장:

- 초기 생산성을 우선하면 `Prisma`

## 인증 방향

초기 후보는 두 가지다.

- 단일 비밀번호 기반 private gate
- Kakao OAuth + 허용 사용자 1명

현재 판단:

- 가장 먼저 MVP를 만들 때는 가능한 한 단순하게 시작하는 편이 낫다.
- 다만 현재 저장소가 이미 Kakao 로그인 흐름을 갖고 있으므로, 완전 제거보다 서버 중심 구조로 재설계하는 선택지가 있다.

권장 순서:

1. MVP 초기에 단순 private gate를 검토
2. 필요 시 Kakao OAuth를 서버 처리 방식으로 재도입

## 업로드 구조

사진/동영상 업로드는 앱 서버를 거쳐 S3에 직접 저장하지 않고, presigned URL 기반 직접 업로드 구조로 간다.

권장 흐름:

1. 사용자가 로그인
2. 업로드 요청
3. 서버가 S3 업로드용 presigned URL 발급
4. 브라우저가 S3에 직접 업로드
5. 업로드 완료 후 서버가 메타데이터 저장

장점:

- 앱 서버 부하 감소
- 대용량 파일 처리에 유리
- 서버 디스크 관리 불필요
- 향후 동영상 업로드 확장에 유리

추가 고려:

- 큰 동영상은 multipart upload 고려
- 파일 형식 제한 필요
- 업로드 크기 제한 필요
- 업로드 실패 재시도 흐름 필요

## 삭제 대상

현재 목표 기준으로 아래 요소는 제거 대상이다.

- `contract/` 전체
- NFT/Mint/Market 관련 화면 흐름
- 클라이언트 중심 Kakao OAuth 처리 방식
- 현재 CRA 전용 빌드 구조
- 불필요한 Recoil persisted auth 구조

## 유지 또는 재활용 가능 요소

완전히 버릴 필요는 없고, 일부는 참고 자산으로 남길 수 있다.

- `Cathy Garden` 브랜딩 이름
- 정원/전원주택 이미지 자산
- 기존 라우팅 구조에서의 개념적 흐름
  - 로그인
  - 메인 페이지

## 마이그레이션 전략

### 전략 원칙

- 한 번에 모든 기능을 옮기기보다, 새 구조를 먼저 세운다.
- 기존 코드 유지보다 새 아키텍처 정합성을 우선한다.
- 제품 목표와 맞지 않는 부분은 과감히 제거한다.

### 권장 순서

1. 테스트 환경 정리
2. 새 `Next.js` 앱 구조 생성
3. 기존 CRA 구조 제거 시작
4. `contract/` 제거
5. 인증 구조 도입
6. S3 업로드 API 도입
7. 보관함 UI 도입
8. 메타데이터 저장 구조 도입
9. 배포 설정

## 디렉터리 방향 초안

전환 후에는 아래와 같은 구조를 목표로 한다.

```text
/
├── app/
├── components/
├── lib/
├── public/
├── tests/
│   ├── agent/
│   └── human/
├── docs/
├── package.json
└── next.config.js
```

예상 역할:

- `app/`: 라우트, 레이아웃, 서버 컴포넌트, API 라우트
- `components/`: 재사용 UI 컴포넌트
- `lib/`: 인증, S3, DB, 유틸
- `tests/agent/`: 에이전트가 관리하는 테스트
- `docs/`: 제품/아키텍처/전환 문서

## 주요 기술 리스크

### 테스트 환경 불안정

현재 저장소의 테스트 환경은 이미 깨져 있다.  
기술 전환 전에 테스트 실행 기반을 먼저 정상화해야 한다.

### 인증 방식 재설계 필요

기존 Kakao 인증은 브라우저 중심 구조이므로, 그대로 유지하면 이후 private archive 구조와 맞지 않는다.

### 미디어 업로드 특성

사진보다 동영상은 용량과 네트워크 실패 가능성이 높다.  
초기부터 대용량 업로드 전략을 염두에 둬야 한다.

### UI 재구성 범위

현재 UI는 제품 목표와 맞지 않는 탭 구조를 가지고 있다.  
부분 수정으로는 한계가 크고, 랜딩부터 보관함 화면까지 새로 설계하는 편이 낫다.

- 마이그레이션 순서가 정리되었다.
