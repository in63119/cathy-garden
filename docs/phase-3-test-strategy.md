# Phase 3. 테스트 전략 수립

## 목적

이 문서는 Cathy Garden 프로젝트를 전환하고 확장하는 과정에서 어떤 테스트를 어떤 방식으로 운영할지 정의한다.

이 프로젝트는 앞으로 구조를 크게 바꿀 예정이므로, 테스트는 단순한 보조 수단이 아니라 전환 안정성을 보장하는 핵심 장치여야 한다.

## 현재 상태

현재 저장소의 테스트 환경은 정상 상태가 아니다.

확인된 문제:

- 기존 테스트는 [src/App.test.tsx](/Users/inbrew/Desktop/cathy-garden/src/App.test.tsx)에 남아 있다.
- 테스트 실행 시 `react-router-dom` 모듈 해석 오류가 발생한다.
- 현재 테스트는 제품 목표와 직접 관련이 없다.
- `tests/agent/` 아래에는 실질적인 테스트 파일이 아직 없다.

즉, 현재 단계에서 우선해야 할 일은 테스트를 많이 쓰는 것이 아니라, 테스트가 신뢰할 수 있는 상태로 먼저 복구되는 것이다.

## 테스트 전략의 기본 원칙

- 모든 코드 변경 후 테스트를 실행한다.
- 에이전트가 작성하거나 수정하는 테스트는 반드시 `tests/agent/` 아래에 둔다.
- 테스트는 제품 목표와 직접 연결되어야 한다.
- 현재 동작을 무의미하게 고정하는 테스트보다, 전환 후에도 가치가 남는 테스트를 우선한다.
- UI 문구보다 사용자 행동과 권한 흐름을 검증하는 테스트를 우선한다.

## 테스트 목표

테스트의 핵심 목표는 다음과 같다.

- 비인가 사용자가 접근하지 못하도록 보장
- 업로드 흐름이 깨지지 않도록 보장
- 업로드 후 목록 반영이 정확한지 보장
- 삭제 후 상태 반영이 정확한지 보장
- 리팩터링 중 핵심 사용자 흐름이 깨지지 않도록 보장

## 테스트 레벨

### 1. 단위 테스트

대상:

- 유틸 함수
- 파일 검증 로직
- 메타데이터 변환 로직
- 경로 생성 로직
- 권한 판별 로직

목적:

- 작고 독립적인 로직을 빠르게 검증
- 예외 케이스를 저비용으로 보장

### 2. 통합 테스트

대상:

- 인증 흐름
- 업로드 요청 API
- 메타데이터 저장 흐름
- 목록 조회와 삭제 흐름

목적:

- 여러 모듈이 함께 연결될 때 기대 동작 보장
- 서버 라우트와 데이터 계층 연결 검증

### 3. UI/라우팅 테스트

대상:

- 보호 페이지 접근 제어
- 로그인 성공/실패 후 이동
- 업로드 화면의 사용자 상호작용
- 목록 화면 렌더링

목적:

- 핵심 사용자 흐름 보장
- 라우팅과 화면 상태의 기본 정합성 확인

### 4. 최소 E2E 테스트

대상:

- 로그인
- 업로드
- 목록 반영
- 상세 보기
- 삭제

목적:

- 실제 사용자 관점에서 가장 중요한 시나리오 보장
- 배포 전 핵심 흐름 최종 확인

## 우선순위

### 1순위

- 테스트 실행 환경 정상화
- `tests/agent/` 기준 확정
- 인증 가드 테스트
- 업로드 API 테스트

### 2순위

- 목록 렌더링 테스트
- 삭제 테스트
- 에러 처리 테스트

### 3순위

- E2E 시나리오 확장
- 시각적 회귀 테스트 필요 여부 검토

## `tests/agent/` 운영 원칙

에이전트가 관리하는 테스트는 모두 `tests/agent/` 아래에 둔다.

예상 구조:

```text
tests/
├── agent/
│   ├── auth-guard.test.ts
│   ├── upload-api.test.ts
│   ├── media-list.test.ts
│   └── delete-media.test.ts
└── human/
```

운영 규칙:

- 새 기능을 만들면 관련 테스트를 함께 추가한다.
- 기존 기능을 바꾸면 관련 테스트를 함께 수정한다.
- 실패하는 테스트를 방치하지 않는다.
- 테스트 파일명은 목적이 바로 드러나야 한다.

## MVP 기준 필수 테스트 항목

### 인증

- 비로그인 상태에서 보호 페이지 접근 시 차단
- 로그인 성공 시 보관함 진입
- 로그인 실패 시 오류 처리

### 업로드

- 허용된 파일 형식 업로드 요청 성공
- 허용되지 않은 파일 형식 차단
- 업로드 URL 발급 실패 처리
- 업로드 완료 후 메타데이터 저장 처리

### 보관함

- 목록 조회 성공
- 빈 목록 상태 렌더링
- 상세 보기 진입
- 삭제 후 목록 갱신

### 권한

- 허용되지 않은 사용자의 요청 차단
- 만료된 세션 또는 무효 세션 차단

## 전환 과정에서의 테스트 정책

현재 구조를 `Next.js`로 전환하는 동안에는 아래 정책을 따른다.

- 제품 목표와 무관한 기존 테스트는 그대로 유지하는 것이 목적이 아니다.
- 필요하면 기존 테스트는 제거하거나 대체한다.
- 기존 CRA 테스트보다 새 아키텍처 기준 테스트를 우선한다.
- 전환 중간에도 최소한의 회귀 방지 테스트를 유지한다.

## 권장 구현 순서

1. 기존 테스트 실패 원인 정리
2. 테스트 실행 환경 정상화
3. `tests/agent/` 기준 첫 테스트 추가
4. 인증 가드 테스트 작성
5. 업로드 API 테스트 작성
6. 목록/삭제 테스트 작성
7. 최소 E2E 시나리오 작성

## 도구 방향

현재 저장소는 CRA 기반 Jest 환경을 사용 중이지만, 전환 이후에는 새 구조에 맞춰 테스트 스택을 다시 정리하는 것이 적절하다.

후보:

- 단위/통합/UI 테스트: `Jest` 또는 `Vitest`
- DOM 테스트: `Testing Library`
- E2E 테스트: `Playwright`

현재 판단:

- 기존 환경을 무리하게 유지하기보다, `Next.js` 전환 시점에 테스트 도구도 함께 재정비하는 편이 낫다.
