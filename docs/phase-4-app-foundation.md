# Phase 4. 새 앱 골격 구성

## 목적

이 문서는 Cathy Garden을 `Next.js` 기반의 새 구조로 전환할 때 필요한 앱 골격을 정의한다.

이 단계의 목표는 기능을 모두 구현하는 것이 아니라, 앞으로 인증, 업로드, 보관함 기능이 안정적으로 올라갈 수 있는 기본 구조를 먼저 정하는 것이다.

## 단계 목표

- `Next.js` 기반 앱 구조를 확정한다.
- `App Router` 기준의 페이지 골격을 정의한다.
- 공통 레이아웃과 공용 컴포넌트 경계를 정한다.
- 이후 인증, 업로드, 보관함 기능이 들어갈 자리를 먼저 만든다.

## 전환 방향

현재 CRA 중심 구조를 유지한 채 부분 수정하는 방식은 지양한다.

대신 아래 원칙으로 새 구조를 만든다.

- 새 앱 구조를 먼저 만든다.
- 공통 레이아웃을 먼저 만든다.
- 핵심 페이지의 빈 골격을 먼저 만든다.
- 이후 기능을 페이지별로 채운다.

## 목표 디렉터리 구조

전환 후 기준 구조 초안:

```text
/
├── app/
│   ├── layout.tsx
│   ├── page.tsx
│   ├── login/
│   │   └── page.tsx
│   ├── library/
│   │   └── page.tsx
│   ├── upload/
│   │   └── page.tsx
│   ├── media/
│   │   └── [id]/
│   │       └── page.tsx
│   └── api/
├── components/
├── lib/
├── public/
├── tests/
│   ├── agent/
│   └── human/
└── docs/
```

## 각 디렉터리 역할

### `app/`

- 라우트 엔트리
- 공통 레이아웃
- 서버 컴포넌트
- API 라우트

### `components/`

- 재사용 가능한 UI 컴포넌트
- 페이지 전용이 아닌 공용 컴포넌트

예상 컴포넌트:

- 헤더
- 네비게이션
- 업로드 카드
- 미디어 그리드
- 미디어 카드
- 빈 상태 화면
- 에러 상태 화면

### `lib/`

- 인증 관련 코드
- S3 관련 코드
- 데이터베이스 관련 코드
- 유틸 함수
- 서버 전용 헬퍼

### `tests/agent/`

- 에이전트가 직접 작성/유지하는 테스트
- 새 구조 기준 테스트 파일 보관

## 라우트 구조

### `/`

역할:

- 제품의 첫 진입 화면
- 브랜드/정체성 표현
- 로그인 또는 보관함 진입으로 연결

화면 목표:

- 전원주택/정원 비주얼 중심
- 제품이 개인용 보관함이라는 점을 짧고 분명하게 전달

### `/login`

역할:

- 인증 진입점
- private gate 또는 Kakao OAuth 시작점

필수 요소:

- 로그인 안내
- 접근 제어 목적 설명
- 로그인 버튼 또는 비밀번호 입력

### `/library`

역할:

- 업로드된 사진/영상 목록
- 서비스의 핵심 메인 화면

필수 요소:

- 목록 영역
- 정렬 기준
- 업로드 진입 버튼
- 빈 상태 처리

### `/upload`

역할:

- 파일 선택 및 업로드

필수 요소:

- 파일 선택 UI
- 업로드 진행 상태
- 실패/성공 피드백

### `/media/[id]`

역할:

- 특정 사진/영상 상세 보기

필수 요소:

- 사진 확대 보기 또는 동영상 재생
- 메타데이터 표시
- 삭제 액션

## 공통 레이아웃 방향

### 루트 레이아웃

루트 레이아웃에서는 다음을 담당한다.

- 기본 메타데이터
- 공통 배경 톤
- 전역 폰트
- 공통 헤더/프레임

### 헤더

헤더는 지나치게 대시보드처럼 만들지 않는다.

역할:

- 브랜드 이름 표시
- 현재 위치 안내
- 주요 이동 링크

예상 항목:

- `Cathy Garden`
- 보관함
- 업로드
- 로그아웃

### 모바일 우선 레이아웃

이 프로젝트는 휴대폰 사용을 우선 고려해야 한다.

원칙:

- 큰 탭/버튼
- 단순한 네비게이션
- 세로 스크롤 중심
- 과도한 다단 레이아웃 금지

## 공용 UI 골격

초기에는 아래 컴포넌트부터 만들 수 있어야 한다.

- `AppShell`
- `SiteHeader`
- `HeroSection`
- `MediaGrid`
- `MediaCard`
- `UploadPanel`
- `EmptyState`

이들은 디자인 시스템과 연결되지만, Phase 4에서는 구조적 자리만 먼저 잡으면 된다.

## 상태 및 데이터 흐름 원칙

새 앱 골격에서는 아래 원칙을 따른다.

- 서버 데이터는 가능한 한 서버 기준으로 다룬다.
- 인증 상태는 브라우저 저장소보다 서버 세션 기준으로 간다.
- 클라이언트 전역 상태는 최소화한다.
- 업로드 상태처럼 화면 상호작용에 필요한 상태만 클라이언트에 둔다.

## API 골격 방향

Phase 4에서 API 전체를 구현할 필요는 없지만, 이후를 위한 위치는 정해 둔다.

예상 API 범위:

- 인증 시작/콜백
- 업로드 presigned URL 발급
- 업로드 완료 처리
- 목록 조회
- 삭제 요청

예상 위치:

- `app/api/auth/...`
- `app/api/upload/...`
- `app/api/media/...`

## 현재 구조에서 제거할 것과 연결할 것

### 제거 대상

- CRA 중심 라우팅
- NFT/Mint/Market 탭 구조
- Recoil persisted auth 구조
- `contract/` 의존 흐름

### 재활용 가능 자산

- `Cathy Garden` 이름
- 정원/전원주택 이미지 자산
- 기존 로그인/메인 페이지라는 개념적 흐름

## 구현 순서

1. `Next.js` 앱 초기화
2. `app/` 라우트 골격 생성
3. 루트 레이아웃 생성
4. 홈/로그인/보관함/업로드/상세 페이지 빈 구조 생성
5. 공통 헤더 및 셸 구성
6. 테스트 가능한 기본 렌더 구조 정리

## 완료 기준

Phase 4가 완료되었다고 볼 수 있는 기준은 다음과 같다.

- 새 디렉터리 구조가 문서화되었다.
- 핵심 라우트 골격이 정의되었다.
- 공통 레이아웃 방향이 정의되었다.
- 공용 컴포넌트 경계가 정의되었다.
- 이후 인증/업로드/보관함 기능이 들어갈 구조가 정리되었다.

## 다음 단계 연결

Phase 4가 끝나면 다음 작업으로 넘어간다.

1. 디자인 시스템 및 브랜딩 구체화
2. `Next.js` 실제 앱 생성
3. 테스트 환경 정상화 및 첫 `tests/agent/` 테스트 추가
